# Version 1.0 Phase 1: Actionable Issue List

## Purpose
Implementation backlog for Phase 1 with concrete acceptance criteria.

## Priority guide
- `P0`: must complete before Version 1.0 release
- `P1`: should complete in Version 1.0
- `P2`: queued for next minor versions

## Issues

### P0

#### V1P1-001 Canonical contract publication
- Type: architecture
- Surfaces: backend, admin, storefront
- Current state: endpoint and module boundaries are split and partly implicit.
- Target state: one signed-off canonical contract for default runtime.
- Acceptance criteria:
  - `docs/v1.0-phase1-contract-catalog.md` exists and is approved.
  - canonical and non-canonical families are explicitly listed.
  - profile/schema policy matrix is documented.
- Suggested commit: `v1.0.0-01-contract-catalog`

#### V1P1-002 Canonical endpoint map lock
- Type: API governance
- Surfaces: backend, admin, storefront
- Current state: teams rely on mixed docs and inferred endpoint usage.
- Target state: single endpoint map with ownership and consumers.
- Acceptance criteria:
  - `docs/v1.0-phase1-canonical-endpoint-map.md` exists and is approved.
  - each endpoint family has declared consumer(s).
  - non-canonical families are excluded from new work.
- Suggested commit: `v1.0.0-02-endpoint-map`

#### V1P1-003 Deprecation plan formalization
- Type: architecture governance
- Surfaces: backend
- Current state: deprecated/transitional scopes are not consistently marked.
- Target state: every non-canonical scope has status and migration path.
- Acceptance criteria:
  - `docs/v1.0-phase1-deprecation-map.md` exists and is approved.
  - deprecated and transitional scopes include target status.
  - removal criteria are defined.
- Suggested commit: `v1.0.0-03-deprecation-map`

#### V1P1-004 Feature availability contract for UI parity
- Type: API + frontend alignment
- Surfaces: backend, admin, storefront
- Current state: storefront has unsupported flows while admin exposes related ops.
- Target state: backend exposes explicit feature availability flags; both UIs consume them.
- Acceptance criteria:
  - one API response includes feature flags for returns/fulfillment/payments availability.
  - storefront and admin conditionally render unsupported flows from same contract.
  - no dead-end CTA remains for unsupported features.
- Suggested commit: `v1.0.0-04-feature-availability-contract`

### P1

#### V1P1-005 OpenAPI deprecation tags and docs sync
- Type: API documentation
- Surfaces: backend
- Current state: deprecated endpoints are not clearly tagged in OpenAPI.
- Target state: OpenAPI marks transitional/deprecated operations.
- Acceptance criteria:
  - deprecated/transitional operations include deprecation metadata.
  - generated API matrix aligns with contract catalog.
- Implementation evidence:
  - `backend/src/main/java/com/noura/platform/config/OpenApiDeprecationConfig.java`
  - `backend/src/test/java/com/noura/platform/config/OpenApiDeprecationConfigTest.java`
  - `docs/v1.0-phase1-deprecation-map.md`
  - `docs/v1.0-phase1-canonical-endpoint-map.md`
- Suggested commit: `v1.0.0-05-openapi-deprecation`

#### V1P1-006 Frontend wrapper cleanup
- Type: frontend refactor
- Surfaces: admin, storefront
- Current state: wrappers include paths not aligned to canonical contract strategy.
- Target state: wrappers grouped by canonical families only.
- Acceptance criteria:
  - admin wrappers reference only canonical families for active flows.
  - storefront wrappers do not call legacy families.
  - dead exports are removed or documented.
- Implementation evidence:
  - `backend/src/main/java/com/noura/platform/controller/EnterpriseInventoryController.java` (`GET /api/v1/admin/inventory/warehouses`)
  - `frontend/admin-dashboard/src/shared/api/endpoints/inventoryEnterpriseApi.js`
  - `frontend/admin-dashboard/src/features/inventory/EnterpriseInventoryPanel.jsx`
  - `frontend/storefront-noura/lib/apiClient.js`
- Suggested commit: `v1.0.0-06-frontend-wrapper-cleanup`

#### V1P1-007 Schema policy guard verification in CI
- Type: DevOps
- Surfaces: backend
- Current state: local profile allows schema drift (`ddl-auto=update`, Flyway off).
- Target state: CI checks prevent non-local unsafe schema policy.
- Acceptance criteria:
  - automated check verifies non-local profiles use migration-first policy.
  - CI fails when production profile uses unsafe schema settings.
- Implementation evidence:
  - `.github/workflows/backend-schema-policy.yml`
  - `backend/scripts/check_schema_policy.py`
  - `backend/src/main/java/com/noura/platform/config/SchemaSafetyStartupValidator.java`
  - `backend/src/test/java/com/noura/platform/config/SchemaSafetyStartupValidatorTest.java`
- Suggested commit: `v1.0.0-07-schema-policy-ci`

#### V1P1-008 Critical path contract tests
- Type: QA automation
- Surfaces: backend, admin, storefront
- Current state: backend tests are stronger than frontend integration coverage.
- Target state: minimum cross-surface contract tests exist.
- Acceptance criteria:
  - auth, cart, checkout, orders, inventory stock list paths are contract-tested.
  - test fixtures assert canonical envelope and error code behavior.
- Suggested commit: `v1.0.0-08-contract-tests`

## Definition of ready for Version 1.0 implementation phase
1. All P0 issues are complete and reviewed.
2. At least 3 P1 issues are complete (`005`, `006`, and one of `007`/`008`).
3. Canonical contract docs are linked from root docs index or release notes.
