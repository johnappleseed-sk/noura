# Version 1.0 Phase 1: Deprecation Map

## Purpose
Define what is deprecated, what remains transitional, and what is canonical for default runtime.

## Status legend
- `canonical`: active and approved for new work
- `transitional`: still active but should be replaced
- `deprecated`: no new feature work; migration only
- `retired`: removed from active runtime

## Deprecation map

| Scope | Current status | Target status | Rationale | Migration path |
| --- | --- | --- | --- | --- |
| `/api/v1/*` platform endpoints | canonical | canonical | active enterprise contract | continue |
| `/api/inventory/v1/*` inventory endpoints | canonical | canonical | active warehouse contract | continue |
| `/api/storefront/v1/*` legacy storefront endpoints | deprecated | retired after migration complete | profile-gated and non-canonical for active apps | migrate all consumers to `/api/v1/*` |
| `com.noura.platform.commerce.*` runtime APIs | deprecated | retired or selectively rebuilt into canonical modules | excluded from default component scan, creates false-positive feature assumptions | rebuild needed capabilities into `com.noura.platform.controller` + active services |
| `/api/v1/inventory/*` platform inventory aliases | transitional | deprecated | overlaps with `/api/inventory/v1/*` | shift admin usage to inventory family only |
| duplicate stock endpoints (`/api/v1/inventory/{variantId}` and `/api/v1/inventory/variants/{variantId}`) | transitional | deprecated single-path alias | duplicate surface area | keep one path, mark the other as compatibility alias |

## Required deprecation mechanics
1. Mark deprecated controllers/services/classes with code annotations and Javadoc deprecation notes.
2. Add a `docs/deprecation` release note entry per version.
3. For API paths, add OpenAPI `deprecated: true` on transitional/deprecated operations.
   - Implemented by backend OpenAPI customizer: `backend/src/main/java/com/noura/platform/config/OpenApiDeprecationConfig.java`.
   - Transitional/deprecated operations also emit:
     - `x-contract-status` (`transitional` or `deprecated`)
     - `x-deprecation-reason` (migration guidance)
4. Add migration owner and target removal version for each deprecated path/module.

## Non-breaking policy for Version 1.0
- Do not remove deprecated endpoints in Version 1.0.
- Keep compatibility behavior stable for active frontend integrations.
- Emit deprecation notices in docs and OpenAPI only.

## Exit criteria to retire deprecated scopes
- No active admin/storefront imports call deprecated endpoints.
- Contract tests pass only against canonical families.
- Operational runbooks and API docs updated.
- Removal plan approved for next major cycle.

## Ownership
- Backend architecture owner: module and endpoint deprecation tags.
- Frontend owners: remove deprecated endpoint wrappers from admin/storefront.
- QA owner: contract regression for canonical-only pathways.
